Pylint & Others

Pylint

In Python language, Pylint is a tool to check for errors in your code. It helps to find possible problems in your codebase. It also gives a quick overview about your complex python code.

Type pylint in the command line prompt:

root@95b7ca75e8d8:/home/myuser# pylint

You have to care about messages of convention, refactor, warnings, errors, fatal after running this command. Some example of the messages:

simplecaesar.py:3:0: W0301: Unnecessary semicolon (unnecessary-semicolon) simplecaesar.py:1:0: C0114: Missing module docstring (missing-module-docstring) simplecaesar.py:5:0: C0103: Constant name "shift" doesn't conform to UPPER_CASE naming style (invalid-name) simplecaesar.py:9:0: C0103: Constant name "encoded" doesn't conform to UPPER_CASE naming style (invalid-name) simplecaesar.py:13:12: C0103: Constant name "encoded" doesn't conform to UPPER_CASE naming style (invalid-name)

Pep-8

Pep-8 is a style guide for Python code. It was written by Guido van Rossum, Barry Warsaw, and Nick Coghlan. Fundamental principle of the Pep-8 is to improve consistency and readability of the codebase in Python language. Readability is very important because a code is read much more often than it is written as Guido van Rossum said. After a while after you wrote a code, you can understand all of the code in a short time if you follow the Pep-8 guide.

How To Choose Names:

While you are trying to understand a code, you have to figure out so many things by looking at the name. Therefore, it is very important to give common styles to your types.

For example: function uses a lowercase word, and words with underscores like “function, my_function”. Class starts with a capital letter and continues with lowercase words.

In addition to convention, giving descriptive names of what the object represents improves readability. Instead of giving the names of “x” and “y” to represent first name and last name for your variables, giving the names of “first_name” and “last_name” improves readability of the code.

Code Layout:

* Black lines improve readability of the code.

* PEP 8 suggests lines should be limited to 79 characters.

* If breaking lines needs binary operation, you should put that before the operation.

* Put 4 consecutive spaces for indentation instead of using tabs.

Comment and Documentation String:

Putting a comment inside of your code helps you to understand the functionality of your code blocks. Use “#” and single space before your explanation. By giving inline comments, be more specific instead of explaining everything. Put """ before and after your explanation for multiline strings.

Linters and Autoformatters:

Linters and autoformatters are tools to help your code to fit the complaint of Pep-8. While Linters shows flag errors and stylistic problems in the code. Autoformatters reformat your code automatically to fit Pep-8 compliant.
